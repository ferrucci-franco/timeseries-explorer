Full Desktop large-file support:
0) Product goal:
   a) Full Desktop must be the runtime that can seriously handle very large local files.
   b) Do not rely on DuckDB-WASM as the main solution for multi-GB files in Full Desktop.
   c) Do not send whole multi-GB files from Electron main to the renderer as Blob/File/ArrayBuffer.
   d) Use native disk access and native DuckDB in Electron/Node for CSV/Parquet.
   e) Treat MAT multi-GB support as a separate later phase unless a partial MAT reader is designed.
1) Current blocker to remove:
   a) `src/app/methods/file-methods.js::_readLocalResultPath` currently does `fetch(...).blob()` and wraps the result in `new File(...)`.
   b) This defeats streaming for multi-GB files and must not be used for the Full Desktop large-file path.
   c) Keep it only for small files or replace Desktop file loading with dataset descriptors.
2) Define a runtime data-source abstraction used by PlotManager:
   a) Keep the current in-memory legacy data object for small files.
   b) Keep DuckDB-WASM lazy data as the Web Light source.
   c) Add a new Desktop native data source for Full Desktop.
   d) The UI/PlotManager should ask the data source for overview/range data instead of assuming every variable has full `.data` arrays.
3) Create an Electron native data engine:
   a) Add an Electron main-process module such as `electron/data-engine.cjs`.
   b) Use the existing `duckdb` Node package, not DuckDB-WASM.
   c) Manage one DuckDB database/connection per app session or a small connection pool.
   d) Store open datasets in a Map keyed by `datasetId`.
   e) Expose IPC methods through preload with contextIsolation on.
4) Add IPC API in Electron:
   a) `omv:dataset-open` with `{ path, preferredFormat? }`.
   b) `omv:dataset-close` with `{ datasetId }`.
   c) `omv:dataset-get-overview` with `{ datasetId, variables?, maxPoints }`.
   d) `omv:dataset-get-range` with `{ datasetId, variables, t0, t1, maxPoints }`.
   e) `omv:dataset-get-source-window` with `{ datasetId, variable, fromX, direction, maxRows, contextRows }` for cursors/min/max/zero-crossing.
   f) `omv:dataset-cancel-query` with `{ datasetId }`.
   g) `omv:dataset-convert-to-parquet` with `{ path, outputPath?, compression?, overwrite? }`.
5) Dataset open behavior:
   a) Detect extension: `.parquet`, `.csv`, `.txt`, `.mat`.
   b) For Parquet, create a DuckDB view over `read_parquet(path)`.
   c) For CSV/TXT, inspect a sample using the existing CSV profile logic or a shared helper.
   d) Use that profile to build `read_csv(...)` options rather than blindly relying on `read_csv_auto`.
   e) Detect the time column using the same semantics as the Web parser.
   f) Return a descriptor, not file contents:
      - `datasetId`
      - `path`
      - `displayName`
      - `format`
      - `columns`
      - `variables`
      - `timeColumn`
      - `timeKind`
      - `timeStart`
      - `timeEnd`
      - `rowCount` if known
      - `warnings`
6) Overview generation:
   a) For Parquet, use DuckDB queries with projection and time ordering.
   b) For CSV, avoid materializing the whole CSV in JS.
   c) Produce at most a fixed number of points per selected/visible variable for initial rendering.
   d) The overview must be enough to populate the variable tree, metadata, and first plot without huge memory growth.
7) Range queries for plotting:
   a) Implement the same bucketed query idea used by `DuckDbSource.getColumnsRange`.
   b) Input: variables + time range + maxPoints.
   c) Output: typed arrays or transferable arrays small enough for Plotly.
   d) Group multiple variables from the same file into one query.
   e) Cancel stale queries when the user keeps zooming/panning.
   f) Cache a small number of recent range queries per dataset.
8) Cursor/source-window queries:
   a) Implement exact raw-window queries around a time point, similar to current WASM `fetchSourceWindow`.
   b) Use it for measurement cursors, next sample, local maxima/minima, and zero crossing.
   c) Keep the result bounded, e.g. 50k rows, not an entire file.
9) CSV-to-Parquet conversion inside Full Desktop:
   a) Move the useful logic from `bench/csv-to-parquet.mjs` into a reusable Electron/Node module.
   b) Add a UI flow when opening a large CSV:
      - "This CSV is very large. Convert to Parquet for faster loading?"
      - Buttons: "Convert and open", "Open directly", "Cancel".
   c) Let the user choose output path or default to same folder.
   d) Show progress/status if DuckDB exposes progress; otherwise show an indeterminate progress with file names.
   e) Open the resulting Parquet automatically after conversion.
   f) Never require the user to run a terminal command for ordinary use.
10) Threshold policy:
   a) Small CSV/MAT can keep using current in-memory parser.
   b) Medium CSV/Parquet can use existing DuckDB-WASM in Web Light.
   c) Full Desktop should use native data engine for CSV/Parquet above a conservative threshold, e.g. 50 MB or when selected through Desktop picker.
   d) For CSV above a larger threshold, e.g. 500 MB, recommend/propose Parquet conversion.
   e) For MAT above a safe threshold, show a clear message until partial MAT support exists.
11) MAT large-file phase, later:
   a) Audit OpenModelica/Dymola MAT file structure used by typical result files.
   b) Determine whether variables can be read independently without loading the whole file.
   c) If yes, design a partial MAT reader with variable/time indexing.
   d) If no, be honest in UI: MAT large-file support is limited compared with CSV/Parquet.
12) Renderer integration:
   a) Update `loadFile/loadFiles` so Desktop path selection can return dataset descriptors instead of File objects.
   b) Add PlotManager support for Desktop-backed lazy datasets.
   c) Reuse the existing lazy refresh path where possible; do not duplicate all plotting logic.
   d) Make file close release native datasets through IPC.
   e) Make session save/restore store enough dataset metadata/path to restore Desktop-backed datasets when possible.

   c) If a query fails, keep the previous plot visible and show a local warning.
   d) If conversion fails, show the DuckDB error in a user-readable way.
